CAREER MODE IMPLEMENTATION STATUS LOG
Updated: 2026-03-05

Status key:
- NOT_STARTED: feature behavior is not implemented.
- PARTIAL_FOUNDATION: reusable engine/app pieces exist, but feature is not implemented end-to-end.
- IMPLEMENTED: feature requirements completed.

F001 | IMPLEMENTED
Notes: Added "Start Career" button on Home. Clicking it sets `gameState.mode` to `career` and navigates to `/career/start`. Added `/career/start` placeholder screen with no debug UI. Added default context field `mode: "sandbox"`.

F002 | IMPLEMENTED
Notes: Added `/career/calendar` placeholder route/screen and centralized `CareerRouteGuard` in `src/routes.jsx`. Guard redirects debug routes (`/team-selection`, `/match`, `/match-sim`) to `/career/start` when `gameState.mode === "career"`. Career mode is re-applied on refresh when path starts with `/career`.

F003 | IMPLEMENTED
Notes: StartCareer now has Team Name input plus Home/Away color pickers with validation: non-empty team name, colors must differ, and contrast ratio threshold check. Continue is disabled until valid and advances to a local next-step placeholder.

F004 | IMPLEMENTED
Notes: StartCareer now runs as a multi-step flow: Team Details -> sequential selection steps (GK, DEF1, DEF2, MID1, MID2, FWD) -> Summary. Draft state stores explicit slots (`GK`, `DEF1`, `DEF2`, `MID1`, `MID2`, `FWD`) and prevents duplicate player picks across slots.

F005 | IMPLEMENTED
Notes: Each draft slot now uses a random 4-candidate queue with one current candidate shown at a time. Added Keep/Skip controls, "Candidate X of 4" display, and forced keep on candidate 4 (Skip disabled). Queue state is stored per slot and persists while moving through steps.

F006 | IMPLEMENTED
Notes: Draft candidate generation is now role-driven with rarity bands. Typical candidates generate in OVR 68-74, rare candidates in OVR 75-80. Rare chance scales by skips per slot: base 5% + 6% per skip, capped at 25%. Attributes are generated by role profile (GK/DEF/MID/FWD) and overall is computed/tuned per preferred role.

F007 | IMPLEMENTED
Notes: Summary screen now shows validated team name, home/away colors, and full chosen starting squad from draft slots. Added Back and Start Career actions. Start Career commits a serializable `career.playerTeam` payload to game context (team identity, colors, baseTier, lineup slots, squad attributes) and continues into the career generation flow.

F008 | IMPLEMENTED
Notes: Added dedicated `/career/loading` screen with spinner and staged generation messages: "Generating teams", "Creating leagues", "Preparing calendar". StartCareer now transitions to loading after confirmation and marks career status as `generating`. Loading uses engine `createTimer` and transitions automatically to `/career/calendar`, updating career status to `ready` with generation metadata.

F009 | IMPLEMENTED
Notes: Added career AI-team generation utility and integrated it into loading completion. Career setup now creates and stores exactly 79 AI teams in `gameState.career.aiTeams` with fields: `id`, `name`, `homeColor`, `awayColor`, `baseTier`, and `teamStrength`. Generation is deterministic from career seed and avoids regeneration if 79 teams already exist.

F010 | IMPLEMENTED
Notes: Added deterministic team-name generator in career team generation utility using seeded pools (prefix/root/suffix combinations) with uniqueness enforcement. Generation reserves the player team name to avoid collisions, guaranteeing unique names across the full 80-team world and unique names for all 79 AI teams.

F011 | IMPLEMENTED
Notes: Extended career AI team generation to produce stronger kit contrast and reduce clashes with player-team colors when available. Added color-contrast checks in the generator and kit selection retries so home/away colors stay visually distinct, while preferring colors that contrast away from player home/away kits.

F012 | IMPLEMENTED
Notes: Added career league composition generator with the exact 10-tier slot matrix and 8 teams per league. Leagues are now created during career loading and stored in `gameState.career.leagues`. Player team is explicitly placed in Tier 1, and AI teams are assigned by `baseTier` according to the required distribution pattern.

F013 | IMPLEMENTED
Notes: Extended `saveGame` utilities with localStorage support (save/load/clear) using the same Base64 format as file saves. GameContext now hydrates from localStorage on boot and normalizes loaded state. Added career autosave on major milestones (`generating`, `ready`) so player team, AI teams, and leagues persist across reloads.

F014 | IMPLEMENTED
Notes: Added season calendar model utility for a fixed 56-day season (2 months, 4 weeks per month, 7 days per week). Season model is created/preserved during career loading and stored at `gameState.career.season`. Career Calendar now renders month/week/day UI from that model and displays 56-day coverage.

F015 | IMPLEMENTED
Notes: Career Calendar now functions as the main hub with dedicated sections for Next Fixture, Team Management link, League Table, Staff link placeholder, and Continue action. Added placeholder routes/screens for `/career/team-management` and `/career/staff`. Continue advances season day state for hub flow while deeper match trigger logic remains for later features.

F016 | IMPLEMENTED
Notes: Added deterministic fixture generation utility for each 8-team league using a single round-robin (7 matchdays, 4 matches per matchday). Fixtures are scheduled only on Saturday/Sunday dates from the season model and stored in `gameState.career.fixtures`. Week 8 is explicitly reserved for a playoff final slot in fixture metadata.

F017 | IMPLEMENTED
Notes: Added day-level league simulation for non-player fixtures when the player has a match that day. On Continue, AI vs AI fixtures for that day are simulated, results are recorded on fixtures, and league tables are recalculated from played results. Player fixture remains unsimulated for later match-trigger integration.

F018 | IMPLEMENTED
Notes: Expanded league table state to fully track and persist `Played`, `Wins`, `Draws`, `Losses`, `Goals For`, `Goals Against`, `Goal Difference`, and `Points` from played fixtures. Calendar league table now renders these values directly, and table sorting uses points then goal metrics for ordering.

F019 | IMPLEMENTED
Notes: Career Calendar Continue now checks the current day for the player fixture; on match day it simulates other fixtures for that day, creates a career match `pendingConfig`, and navigates to `/match`. Match completion now writes the played result back onto the correct career fixture, recalculates league tables, advances/completes the season day, clears active career match state, and returns to `/career/calendar`.

F020 | IMPLEMENTED
Notes: Added end-of-season resolution flow in Career Calendar. On/after final Sunday, playoff finals are generated from top-2 league standings, non-player finals are simulated, player finals trigger the normal match flow, and once all playoff finals are played the season is resolved into a history snapshot and a new season is created (season number increments, calendar resets, league fixtures/tables regenerated).

F021 | IMPLEMENTED
Notes: Added season-to-season promotion/relegation restructuring with a persistent team pool. One team is promoted/relegated between adjacent tiers (top tier cannot promote out, bottom tier cannot relegate down), using playoff-final winner where available with table fallback. Next-season leagues, fixtures, and tables now use the updated tier memberships, and teams store `currentTier` for continuity.

F022 | IMPLEMENTED
Notes: Added relegation tracking in career state with `totalRelegations` and `consecutiveRelegations`, including bottom-tier last-place "relegation strike" handling while keeping the team in the bottom league. End-of-season resolution now evaluates game-over rules (2 consecutive relegations or 5 total relegations), sets career `status: game_over` with reason/timestamp, and blocks further season progression in Calendar. Added autosave milestone support for `game_over`.

F023 | IMPLEMENTED
Notes: Added persistent career victory tracking with flags `wonTopLeague` and `wonChampionsCup` plus a `victoryProgress` object. End-of-season resolution now marks `wonTopLeague` when the player team wins Tier 1, preserves cross-season progress, and computes combined win state (`isCareerWon`) for the two-condition victory requirement. Champions Cup flag is retained for future cup integration. Calendar now displays both victory flags, and autosave milestones include victory-progress changes.

F024 | IMPLEMENTED
Notes: Added tier-based cup eligibility model with exact mapping: Tier 10 = no cup, Tiers 9-6 = Swap Cup, Tiers 5-2 = Super Swap Cup, Tier 1 = Champions Cup. Eligibility is persisted in `career.cupEligibility` for all teams, computed during initial career generation and recomputed on season transitions after promotion/relegation. Team cup metadata is also attached to team records for future cup systems, and Calendar now displays the player team's current cup eligibility.

F025 | IMPLEMENTED
Notes: Added Swap Cup structure generation as a 32-team knockout sourced from exactly four leagues (tiers 6-9). The structure includes seeded entrants, full round scaffolding (R32, R16, QF, SF, Final), and placeholder match nodes for progression. Swap Cup state is now persisted at `career.cups.swapCup` during initial career generation and regenerated each season transition.

F026 | IMPLEMENTED
Notes: Added Champions Cup structure generation as a 32-team knockout with 31 generated foreign clubs and 1 domestic entry slot. The structure now includes seeded entrants and full round scaffolding (R32, R16, QF, SF, Final) under `career.cups.championsCup`. Champions Cup state is generated during career loading and regenerated each season transition.

F027 | IMPLEMENTED
Notes: Added cup scheduling metadata to knockout structures. Cup rounds are scheduled for Wednesday every other week (weeks 1/3/5/7) with final on the last Wednesday for Swap Cup. Champions Cup follows the same every-other-week Wednesday cadence for early rounds with final explicitly scheduled on the last Sunday. Schedule data is persisted in each cup's `roundSchedule` and mirrored onto match nodes.

F028 | IMPLEMENTED
Notes: Added cup draw event workflow with a dedicated `/career/cup-draw` screen that reveals pairings for the next pending cup round. Calendar Continue now checks Friday draw windows and routes to the draw screen before day progression. Confirming the draw marks the round as revealed (`drawnRounds` + history) and returns to Calendar; draw state is persisted in career data.

F029 | IMPLEMENTED
Notes: Added cup match-day flow into Career Calendar. On scheduled cup days, non-player cup ties are now simulated and written into cup state, while player cup ties create a career `pendingConfig` and route into `/match`. Match completion now supports `competitionType: "CUP"` and writes results back into the correct cup round/match, resolves knockout winners (including penalty tiebreak for draws), propagates winners into the next round bracket, updates cup status/winner, and persists cup progress in career state.

F030 | IMPLEMENTED
Notes: Replaced Career Team Management placeholder with a functional management screen. It now shows the full player squad with role-relevant ratings, allows formation selection from all four required shapes (1-2-2, 2-2-1, 2-1-2, 1-3-1), and supports explicit slot-by-slot position assignment with duplicate prevention and lineup validation. Saved setup is persisted to `career.playerTeam.matchSetup` and integrated into career match generation so league and cup matches use the selected formation and lineup.
